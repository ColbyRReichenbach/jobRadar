# AI Feature Production Spec: Final Verified Architecture

Date: 2026-05-11
System: AppTrail / JobRadar AI platform
Source documents: `ai-feature-production-spec.md` and `ai-feature-production-spec-gap-review.md`
Evidence posture: current-state claims in this document are limited to code-inspected behavior and local artifact counts that were verified during review. Production targets are explicitly marked as proposed architecture.

## Purpose

This document explains how the current AI features work, where the original spec overclaimed, and what architecture should guide the system toward production-grade AI.

The core production thesis remains:

```text
User or system signal
  -> classify intent
  -> normalize entities
  -> dedupe against user state
  -> retrieve grounded evidence
  -> generate or recommend only from evidence
  -> validate output
  -> write model/eval/governance artifacts
  -> collect feedback
  -> feed labels/evals back into improvement loops
```

The important correction is that every stage needs a promotion checkpoint. The system should not move from rules to ML, from lexical retrieval to RAG, or from retrieval to agents because the next layer is more sophisticated. It should move only when data volume, eval results, risk controls, and user outcomes justify the switch.

## Product Scope

AppTrail / JobRadar is not one AI feature. The product spans:

- Job/application tracking from dashboard and browser extension captures.
- Gmail ingestion, classification, application linking, interview suggestions, and network-contact suggestions.
- Source intelligence for job URLs, private links, career pages, ATS providers, and source verification.
- Search documents used by dashboard search and Copilot retrieval.
- Copilot as a read-only assistant over user-owned records.
- Resume parsing and tailoring.
- Radar and research workflows that fetch sources, extract evidence, write reports, derive actions, and emit alerts.
- AI governance primitives: model-call ledger, safety decisions, experiments, promotion reports, model cards, feedback, and generated eval artifacts.

The production system must treat these as one AI platform. Gmail, Radar, Copilot, extension capture, and resume tailoring should share entity normalization, dedupe, evidence, retrieval, validation, safety, feedback, and governance infrastructure.

## Evidence And Verification Reality

### Runtime Counts

The original spec included a "recent local database snapshot" table. That table should not be repeated as a verified fact in final materials until it is regenerated by a reproducible script.

During review:

- `DATABASE_URL` and `NEON_DATABASE_URL` were not set in the shell.
- `apptrail.db` was an older minimal SQLite database with only a small number of rows.
- `apptrail-local.db` had many product tables but zero rows for checked runtime tables and lacked several newer AI/search tables.

Final requirement:

```text
Runtime counts must come from a generated artifact that includes:
  database source label
  migration version
  git SHA
  query timestamp
  table counts
  missing-table warnings
```

Local database counts are useful for interview context, but they are not production quality metrics.

## Current Verified System Posture

Verified current behavior:

- Gmail classification is a hybrid deterministic NLP system with optional LLM adjudication only in `hybrid` mode.
- The default env example is `GMAIL_CLASSIFIER_MODE=hybrid_dry_run`; dry-run does not call the model.
- Gmail preflight performs prompt-injection checks, redaction, prompt-size checks, and prompt leak checks before LLM adjudication.
- Gmail sync extracts raw URLs before body parsing and stores sanitized/source links after classification.
- Search is user-scoped and source-level, not chunk-level.
- The default search backend is SQL `LIKE` retrieval with hand scoring.
- The OpenSearch adapter is a placeholder and raises unavailable errors without a concrete client.
- Copilot retrieves `SearchDocument` context, builds a retrieved-context-only prompt, validates citation IDs, stores assistant messages, and supports thumbs feedback.
- Copilot validates that cited document IDs were retrieved for the user; it does not prove semantic support for every claim.
- Resume tailoring sees current resume text, job description, target company/role, and parsed skills. It does not use project/repo evidence.
- Radar has a graph pipeline with persisted steps, source/evidence/report persistence, cost tracking, and a verifier node.
- Radar public web search currently uses broad web search/fetching rather than source-registry-first retrieval.
- Job/contact/interview duplicate checks exist, but they are endpoint-specific rather than unified behind a shared action-candidate layer.
- Source intelligence has useful primitives: `CompanyJobSource`, `JobPosting`, `SourceDiscoveryEvent`, `ApplicationSourceLink`, source verification, and ATS/source adapters.

## Production Principles

### 1. Separate Current State From Proposed Architecture

Use these terms consistently:

- **Implemented**: code path exists and was verified.
- **Scaffolded**: model/table/service exists but is not fully enforced or not connected end-to-end.
- **Proposed**: target architecture only.
- **Unverified**: claim depends on unavailable runtime data, external systems, or unreviewed provenance.

### 2. Separate Classification Confidence From Action Confidence

An email can be correctly classified while the recommended action is still unsafe, duplicate, noisy, or insufficiently supported.

```text
classification confidence != action confidence
```

Action confidence requires:

- Entity resolution.
- Evidence sufficiency.
- Duplicate checks.
- Side-effect policy.
- User confirmation where appropriate.

### 3. Ground Claims By Risk Level

"Every AI-generated claim should be traceable to evidence" needs scope boundaries.

Must cite or validate:

- Job status.
- Contact identity.
- Interview details.
- Salary and compensation claims.
- User work history.
- Resume skills and project claims.
- Company/research findings.
- Action-triggering recommendations.

Should cite when possible:

- Summaries.
- Recommendations.
- Radar brief conclusions.

Need not cite:

- UI copy.
- Formatting text.
- Neutral acknowledgements.
- Purely stylistic resume edits that add no new factual claim.

### 4. Use Source Registry Before Agents

For Radar and job discovery, the production path should be:

```text
verified sources and adapters first
  -> structured records
  -> evidence chunks
  -> retrieval
  -> agents only for bounded fallback
```

Agents can help with exploration and fallback, but production quality comes from source quality, provenance, and repeatable retrieval.

### 5. Promote Systems Only With Checkpoints

No model, prompt, retriever, source strategy, or action policy should be promoted from anecdotal examples alone.

Each promotion should include:

- Dataset version and provenance.
- Real vs synthetic metrics separated.
- Git SHA.
- Prompt/model/retriever version.
- Cost and latency impact.
- Safety result.
- Regression examples.
- Known limitations.
- Promote, reject, rollback, or shadow-more decision.

## Shared Production Architecture

### Shared Components

| Component | Responsibility | Current status |
| --- | --- | --- |
| Intent classifier | Route user/system input into type/subtype/action intent. | Partly implemented for Gmail; Copilot router evals exist but production routing is limited. |
| Entity normalizer | Canonicalize job URLs, company names/domains, contacts, emails, interviews, skills, source docs. | Implemented in pockets; not unified. |
| Dedupe gate | Prevent repeated jobs, contacts, interviews, alerts, reports, recommendations. | Endpoint-specific; shared gate proposed. |
| Evidence store | Persist documents, chunks, citations, metadata, provenance, ownership. | `SearchDocument`, Radar source/evidence tables, job-source records exist; chunked RAG proposed. |
| Hybrid retriever | Combine lexical, vector, metadata filters, freshness, trust, reranking. | Proposed; current search is source-level lexical. |
| Generation layer | Produce structured outputs with evidence IDs and validation constraints. | Implemented in pockets. |
| Validation layer | Reject unsupported claims, unsafe output, malformed actions, cross-user leakage. | Partly implemented; claim-level validation proposed. |
| Feedback layer | Capture accept, dismiss, thumbs, corrections, edits, labels. | Tables/endpoints exist in several areas; sparse data. |
| Eval layer | Run offline and online evals before promotion. | Scripts and artifacts exist; enforcement proposed. |
| Artifact registry | Persist eval runs, model/prompt/retriever changes, dataset versions, decisions. | Scaffolded. |
| Observability layer | Track cost, latency, model, prompt, fallback, safety decisions, quality. | Scaffolded through model/safety ledgers and metrics. |

### Shared Action Model

The current system creates alerts and recommendations directly in multiple paths. Production should introduce an action-candidate layer before notifications or side effects.

```text
Classified signal
  -> ActionCandidate
  -> EntityNormalizer
  -> DedupeGate
  -> PolicyGate
  -> Alert, suppression, link, review, or mutation
```

Proposed model:

```text
ActionCandidate
  id
  user_id
  source_type
  source_id
  action_type
  target_entity_type
  target_entity_id
  target_fingerprint
  dedupe_key
  duplicate_type
  duplicate_matches_json
  policy_decision
  status
  confidence
  requires_confirmation
  evidence_json
  created_at
  updated_at
```

Recommended statuses:

- `proposed`
- `suppressed_duplicate`
- `linked_existing`
- `pending_review`
- `accepted`
- `dismissed`
- `expired`
- `failed_validation`

Stable dedupe key:

```text
user_id + action_type + target_entity_type + target_fingerprint
```

Examples:

```text
user123:add_job:job_url_hash:abc
user123:add_contact:email:jane@company.com
user123:add_interview:application_id+datetime+interviewer_email
user123:radar_report:profile_id+topic_hash+week_bucket
```

## Feature 1: Gmail Classification

### Implemented Current State

Primary implementation areas:

- `backend/services/email_classifier.py`
- `backend/services/gmail_intelligence/orchestrator.py`
- `backend/services/gmail_intelligence/feature_extractor.py`
- `backend/services/gmail_intelligence/scorer.py`
- `backend/services/gmail_intelligence/classifier.py`
- `backend/services/gmail_intelligence/preflight.py`
- `backend/services/gmail_intelligence/adjudicator.py`

Current flow:

```text
Gmail message
  -> normalize email
  -> extract lexical/sender/link/recruiting features
  -> score route and subtype
  -> deterministic classification
  -> optional LLM adjudication only in hybrid mode and ambiguous cases
  -> persist EmailEvent if storage policy allows
  -> store sanitized links
  -> index SearchDocument
  -> create alerts / suggestions where applicable
```

Strengths:

- Good cold-start strategy without needing many labels.
- Explainable feature and scoring path.
- Route-first architecture reduces lifecycle-category confusion.
- LLM calls are gated by ambiguity and preflight.
- Prompt-injection and privacy checks exist before LLM adjudication.
- Audit and label-queue tooling exists.

Current limitations:

- Default `hybrid_dry_run` does not call the LLM.
- Route/subtype/decision metadata is returned by the classifier but not fully persisted on `EmailEvent`.
- The classifier candidate receives subject/body/sender/sender_email; raw extracted URLs are stored later but do not directly inform classifier features.
- Opportunity-discovery and action-review routes are intentionally not stored as email events yet.
- Thresholds are hand-tuned and not calibrated.
- Real labels are still too sparse for deep ML or fine-tuning.
- The classifier is not a trained vectorizer or supervised model.

### Promotion Checkpoints

Gmail should progress in stages:

```text
rules and route-first classifier
  -> real-label audit
  -> TF-IDF/logistic regression shadow lane
  -> calibrated route/subtype baseline
  -> optional heavier model only if simpler models fail on measured cases
```

| Stage | When to use | Switch criteria |
| --- | --- | --- |
| Deterministic route-first classifier | Current cold-start mode. | Keep until real labels cover the main routes and failure modes. |
| TF-IDF/logistic regression shadow model | Once there are roughly 300-500 real labeled emails and at least 30-50 examples for major routes. | Promote only if held-out real route accuracy improves without increasing high-risk false positives. |
| Calibrated route/subtype model | Once labels support confidence calibration by route. | Use separate confidence bands for route, subtype, and review queues. |
| More sophisticated model or fine-tune | Only after simpler baselines plateau and there are thousands of real labeled examples. | Require documented failure modes, cost/latency budget, safety review, and promotion report. |

Promotion metrics:

- Route accuracy.
- Subtype accuracy.
- Precision/recall by class.
- False positive job-relevant rate.
- False negative interview/action-item rate.
- LLM call rate and cost per 1,000 emails.
- High-risk false positive action rate.
- User correction and dismissal rates.

Production requirements:

- Persist classifier traces.
- Track threshold version.
- Use active learning queues for ambiguous and high-impact messages.
- Separate classification confidence from action confidence.
- Never auto-mutate application state unless evidence and policy gates pass.

## Feature 2: Actions And Notifications

### Implemented Current State

Current behavior:

- Gmail sync creates `EmailEvent` rows for stored classifications.
- Relevant stored messages can create alerts.
- Conversation emails can produce network-contact suggestions.
- Interview request emails appear in `/api/interview-suggestions`.
- Accepted interview suggestions create `Interview` rows.
- Calendar sync can identify interview-like events.
- Job, contact, and interview duplicate checks exist in endpoint-specific paths.

Strengths:

- Useful user workflows already exist.
- Interview suggestions require user confirmation.
- Contact suggestions require review/acceptance.
- Duplicate interview detection exists for scheduled time plus interviewer email.

Current limitations:

- No unified `ActionCandidate` model/service.
- Dedupe is scattered across jobs, contacts, links, interviews, alerts, and Radar recommendations.
- Alerts have no general dedupe key or action-candidate link.
- Correct classifications can still create duplicate or noisy user-facing behavior.

### Production Target

Centralize action generation:

```text
EmailEvent / ExtensionEvent / RadarFinding / CopilotRequest
  -> ActionCandidateExtractor
  -> EntityNormalizer
  -> DedupeGate
  -> PolicyGate
  -> NotificationOrSuppression
```

Action types:

- `add_job_to_pipeline`
- `link_email_to_application`
- `add_network_contact`
- `link_email_to_contact`
- `schedule_interview`
- `follow_up_with_contact`
- `update_application_status`
- `review_radar_opportunity`
- `tailor_resume_for_application`

Action evals:

- Exact entity match accuracy.
- Duplicate suppression precision.
- False suppression rate.
- Required-field extraction accuracy.
- Side-effect policy accuracy.
- Accept/dismiss rate by action type.
- Repeated notification count per entity per week.

## Feature 3: Search And Retrieval

### Implemented Current State

Primary files:

- `backend/services/search/documents.py`
- `backend/services/search/indexer.py`
- `backend/services/search/backends/postgres.py`
- `backend/services/search/backends/opensearch.py`

Current flow:

```text
source record
  -> SearchDocumentInput
  -> normalize title/subtitle/body/keywords into search_text
  -> user-scoped SQL LIKE search by query terms
  -> manual scoring by title/subtitle/body hits
```

Strengths:

- User-scoped retrieval.
- Simple portable SQL backend works locally and in CI.
- Email documents intentionally avoid indexing unlimited raw body text.
- Search powers Copilot context and fallback search.

Current limitations:

- No chunking.
- No embeddings or vector search.
- No BM25.
- No reranking.
- No query decomposition.
- No retrieval trace storage.
- OpenSearch adapter is a placeholder, not a working production backend.
- Search corpus is small and should not be treated as mature RAG.

### Production Target

```text
query
  -> intent/query decomposition
  -> metadata filters by user/source/date/entity
  -> keyword retrieval
  -> vector retrieval
  -> merge candidates
  -> rerank
  -> evidence packaging
  -> retrieval trace
```

Implementation path:

1. Keep Postgres lexical search as fallback.
2. Add `DocumentChunk`.
3. Add chunking and indexing.
4. Add retrieval traces.
5. Add embeddings for chunks.
6. Add BM25 or OpenSearch lexical scoring.
7. Add hybrid merge and reranking.
8. Gate high-risk generation on retrieval evals.

Retrieval metrics:

- Recall@k.
- MRR.
- nDCG.
- Citation precision.
- Empty-result correctness.
- Query latency.
- Cross-user isolation.

```text
user question
```

Strengths:

- User-scoped context.
- Read-only prompt contract.
- Citation-ID validation.
- Search fallback path exists.

Current limitations:

- Retrieval quality is limited by source-level lexical search.
- Citation validation proves IDs were retrieved, not that each claim is supported.
- No strong production intent router yet.
- Suggested actions are read-only and not connected to a shared action-candidate layer.
- Local feedback volume was not verified as populated.

### Production Target

```text
message
  -> intent router
  -> retrieval plan
  -> source-specific retriever
  -> answer generator
  -> claim extraction
  -> citation/support validator
  -> optional ActionCandidate proposal
  -> user confirmation
```

Rules:

- Default to read-only.
- Cite evidence IDs for factual claims.
- Abstain when evidence is insufficient.
- Treat retrieved text as untrusted data, not instructions.
- Never expose cross-user data.
- Do not directly mutate state until action policy and dedupe are centralized.

Copilot evals:

- Router accuracy.
- Retrieval recall@k.
- Groundedness.
- Citation validity.
- Unsupported claim rate.
- Abstention quality.
- Safety refusal correctness.
- Feedback reward trend.

## Feature 5: Resume Tailoring

### Implemented Current State

Primary files:

- `backend/services/resume_parser.py`
- `backend/services/resume_tailor.py`
- `backend/main.py`
- `backend/models.py`

Current flow:

```text
resume text/PDF
  -> parse into structured fields or deterministic fallback
  -> job description
  -> LLM rewrites resume from original resume + JD
  -> reject if unverified new tech skills appear
  -> store ResumeDraft where endpoint flow uses drafts
```

Strengths:

- Clear privacy/consent boundary.
- Deterministic fallback extraction.
- Prompt forbids invented experience.
- Validator blocks some unverified skill additions.

Current limitations:

- Model only sees the current resume and job description.
- It cannot know real project details that the user did not provide.
- It cannot reliably turn the same project into a different domain angle without evidence.
- No project/repo ingestion.
- No searchable user knowledge profile.
- No bullet-level evidence citations.
- No resume-specific hallucination eval.

### Production Target, Phased

Do not start with repo zip ingestion. That is too broad for near-term production.

Recommended phases:

1. Manual project facts: user enters project facts, links, and evidence snippets.
2. Resume/JD requirement parser and evidence matcher.
3. Bullet generator with evidence IDs.
4. Secret-scanned README/config ingestion only.
5. Optional deeper repo summarization after allowlists, size limits, binary/vendor exclusions, and evals exist.

Final target:

```text
user uploads/profile artifacts
  -> parse and normalize artifacts
  -> create searchable user knowledge documents
  -> job description parser classifies target role
  -> retrieve relevant evidence
  -> draft tailored resume sections
  -> validate factual claims against evidence
  -> produce keyword/evidence coverage report
  -> user edits become feedback
```

Production rule:

```text
No evidence, no new factual claim.
```

Resume evals:

- JD requirement extraction accuracy.
- Evidence recall@k for each requirement.
- Unsupported claim rate per bullet.
- Skill hallucination rate.
- Supported keyword coverage.
- Human edit distance.
- User accept/regenerate/edit rate.

## Feature 6: Radar

### Implemented Current State

Primary files:

- `backend/tasks/run_research_radar.py`
- `backend/services/research_radar/graph.py`
- `backend/services/research_radar/nodes/*`
- `backend/services/research_radar/llm.py`
- `backend/services/opportunity_radar/*`
- `backend/services/job_sources/*`
- `backend/services/source_intelligence/*`

Current Radar/research graph:

```text
load_tracker_context
  -> normalize_research_brief
  -> validate_brief
  -> plan_research_tasks
  -> run_search_tasks
  -> fetch_documents
  -> extract_evidence
  -> dedupe_and_rank_evidence
  -> build_report_diff
  -> write_report
  -> derive_report_actions
  -> verify_report
  -> persist_report
  -> emit_alerts
  -> schedule_next_run
```

Strengths:

- Strong orchestration shape.
- Research runs and steps are persisted.
- Evidence/source/report tables exist.
- Cost, token, prompt version, and failure tracking exists.
- Verification node exists.
- Radar already thinks in terms of evidence and reports more than other features.

Current limitations:

- Public web retrieval is broad and brittle.
- Radar search/fetch currently behaves more like public web research than source-registry-first research.
- Existing job-source intelligence is not yet the primary Radar source layer.
- Evidence ranking is not yet a robust deterministic trust/freshness/entity-match model.
- Verification is report-level or run-level, not true claim-by-claim support validation.
- Radar actions are not routed through a shared `ActionCandidate` and dedupe layer.

### Source-Registry-First Architecture

Radar is harder than Gmail because there is no bounded inbox corpus and public web evidence is noisy. Production Radar should not be "scrape the web and RAG over whatever was found." It should be:

```text
user research profile
  -> source bundle selector
  -> verified company/job sources first
  -> structured provider adapters and known endpoints
  -> persisted source/job/evidence records
  -> chunked retrieval over trusted evidence
  -> report generation with citations
  -> claim-level verification
  -> ActionCandidate proposals
  -> dedupe and policy gate
```

The codebase already has useful primitives:

- `CompanyJobSource`
- `JobPosting`
- `SourceDiscoveryEvent`
- `ApplicationSourceLink`
- `SourceVerificationRun`
- Provider adapters for Greenhouse, Lever, Ashby, Workable, SmartRecruiters, Workday, structured data, and partial/needs-review providers.

Production Radar should merge the research graph with this source-intelligence layer.

### Recommended Source Priority

| Priority | Source type | Use |
| --- | --- | --- |
| 1 | User-owned records | Applications, captured jobs, Gmail events, contacts, saved companies, company visits. |
| 2 | Verified source registry | `CompanyJobSource` rows with verified public access and recent verification. |
| 3 | Provider adapters | Greenhouse, Lever, Ashby, Workable, SmartRecruiters, Workday, structured data, where public access works. |
| 4 | Persisted `JobPosting` records | Structured job evidence with dedupe keys, descriptions, dates, canonical URLs. |
| 5 | Structured public pages | JSON-LD `JobPosting`, public career pages, engineering blogs, company pages. |
| 6 | Search provider fallback | Used only when direct and verified sources do not answer the need. |
| 7 | Agentic browsing | Last resort with domain allowlists, budgets, robots/terms checks, and evidence quality gates. |

### Job Site Limitations

Actual limitations:

- Many sites hide provider identifiers behind redirects or client-side routing.
- Some require several clicks or filters before the useful job page appears.
- Some listings are client-rendered and do not expose stable structured data.
- Some sources rate-limit or block automated access.
- Some require credentialed access.
- Some sources may have unacceptable terms risk for automated scraping.
- Search results are unstable and can return stale, duplicate, or irrelevant pages.
- Broad web retrieval cannot reliably prove a company has no matching jobs.

Recommended source statuses:

- `verified`
- `pending`
- `needs_review`
- `stale`
- `blocked`
- `credentialed_required`
- `robots_disallowed`
- `rate_limited`
- `unsupported_provider`
- `company_identity_conflict`

Radar should distinguish:

```text
verified Greenhouse source returned no matching roles today
```

from:

```text
broad search failed to find jobs
```

Those are not the same evidence state.

### What Engineering Can Solve

Solvable:

- Provider-specific source parsing instead of generic scraping.
- Verified source registry with trust, freshness, robots/terms risk, and failure reason.
- Scheduled source verification and stale-source demotion.
- Job posting persistence with provider job ID, canonical URL, title, company, location, description hash, and active status.
- Dedupe across extension captures, Gmail links, Radar findings, and persisted postings.
- Source-bundle selection per Radar profile.
- Retrieval traces for every Radar report and Copilot answer.
- Claim-level citation coverage and unsupported-claim metrics.
- Agentic browser fallback only for allowlisted domains and bounded tasks.
- Human review queues for source conflicts and high-value uncertain findings.

Not fully solvable:

- Complete coverage of every job site.
- Reliable scraping of credentialed or anti-automation systems.
- Proving absence of jobs from public web evidence.
- Perfect extraction from arbitrary client-rendered career pages.
- Truth guarantees when source evidence is stale, contradictory, or incomplete.

### Radar Production Controls

- Source registry with trust tiers.
- Fetch budget per run.
- Freshness window by source type.
- Deduped source URLs.
- Claim-level evidence IDs.
- Report verifier that checks each claim has supporting evidence.
- Alerts capped and deduped by profile/topic/entity/time bucket.
- Action candidates instead of direct duplicate-prone recommendations.

Radar evals:

- Source precision.
- Source freshness.
- Duplicate source URL rate.
- Evidence support score.
- Claim citation coverage.
- Unsupported claim rate.
- Report usefulness rating.
- Action acceptance rate.
- Alert fatigue rate.

## Feature 7: Browser Extension Job Capture

### Implemented Current State

Current behavior:

- The extension can send job/application data into the pipeline.
- Backend job creation checks hard duplicates by normalized job URL.
- Duplicate check endpoint returns hard, soft, or none.
- Source links are sanitized and stored with hashes.
- Safe public links can contribute to source discovery when consent exists.

Current limitations:

- Extension capture, Gmail classification, and Radar can independently surface the same job.
- Dedupe behavior is not centralized.
- Extension action notifications are not unified with Gmail/Radar action dedupe.

### Production Target

```text
extension capture
  -> normalize job URL/company/role
  -> check hard duplicate
  -> check soft duplicate
  -> create/link application
  -> suppress duplicate notifications
  -> index resulting application/source evidence
```

Important rule:

If the user already captured a job through the extension, a later Gmail email or Radar finding about the same job should link to the existing application, not create a new-job notification.

## Interview Narrative

Concise explanation:

> The current system started with cold-start NLP constraints, so Gmail classification is a deterministic route-first classifier with optional preflight-gated LLM adjudication rather than a trained model. That gives explainability and control while we collect labels. The production direction is to turn every AI feature into an evidence-grounded pipeline: classify intent, normalize entities, dedupe actions, retrieve user-scoped evidence, generate structured outputs, validate citations, and collect feedback for evals.

Key production claim:

> Classification alone is not enough. In a production assistant, especially one that can recommend actions, the system must resolve entities, check duplicates, verify evidence, and require confirmation before side effects.

Radar framing:

> Radar should become a source-registry and evidence-quality system first, an agentic research system second. Agents are useful for bounded fallback, but production quality comes from verified sources, structured records, provenance, retrieval traces, and claim-level verification.

Gmail promotion framing:

> We should switch from rules to TF-IDF/logistic regression only when real labels are sufficient for held-out evaluation and calibration. We should switch to heavier models only when simpler baselines fail on measured cases and the promotion artifact shows better quality without worse action risk, cost, latency, or safety.

Bottom line:

```text
Current system:
  strong beta scaffold with deterministic Gmail NLP,
  source-level lexical retrieval,
  read-only Copilot,
  early Radar graph orchestration,
  source-intelligence primitives,
  and governance scaffolding.

Production target:
  shared action candidates,
  entity normalization,
  dedupe gates,
  source-registry-first Radar,
  chunked hybrid retrieval,
  claim-level validation,
  reproducible eval artifacts,
  and real feedback-driven promotion.
```
